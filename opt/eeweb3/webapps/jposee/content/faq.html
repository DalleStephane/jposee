1         User Interface

1.1        Transaction Tab

1.1.1          What is the meaning of each of the columns displayed on the main screen of the transaction tab?

The following table summarizes the meaning of each column presented:

ID

Unique, auto-generated 'identity' value generated by your database management system and maintained by OLS.Switch as the internal transaction identifier.

Date

Date and time the transaction was processed by OLS.Switch

Store

Four-digit store identifier from which the transaction originated

Term

Four-digit terminal identifier – within the Store – from which the transaction originated

Employee

The employee (ID number and related name from cardholder database table) who initiated the transaction

RC

The Response Code (e.g., AA, EX, ND) passed back to the store system

Message

The “Alpha Display Message” passed back to the store system

M

The “Mode” – as in the Account Entry Mode used to enter the Employee’s identify during the transaction; ‘S’ indicates the scanner was used to read the card; ‘T’ indicates the employee typed in their Social Security Number on the keypad provided

Dur

The duration of the transaction, i.e., how long – in milliseconds – it took OLS.Switch to process the transaction, from the moment it received it to the instant it hands the response back to the TCP/IP communications stack.

Out

The number of transactions outstanding on the request queue at the moment the transaction is received by OLS.Switch.

1.1.2          How is the transaction data sorted?  Can I choose a different sort order?

By default, the information in the Transaction tab is presented in order of ascending ID.  The current order of the data set is signified by a white arrow (ascending ▲; descending ▼) in the currently defined sort column.  By clicking on the appropriate column header, you can re-order the presentation data to be sorted by that column.  Currently, you can elect to sort the data by ID, Date, Store, Term, Employee, RC, Message, Dur and Out values.  Clicking your chosen sort column a second time will change the direction of the sort from ascending to descending or vice versa.

Implementation Notes: 

    •  Choosing ‘Term’ as the sort column will cause the data to be ordered by Store + Term.  [It makes no sense to present, for example, all transactions from terminal ID 0001 across all stores.]

1.1.3          How often is the data in the Transaction Tab refreshed?

By default, the Transaction screen requires you to select the “reload” function of your browser screen to refresh the data.  However, you may opt to set your browser to refresh every 30 seconds.  To do so, click on the “Auto-Refresh” link in the bottom right-hand corner of your browser screen.  Each refresh will include references to all new transactions that have occurred since the previous refresh.

1.1.4          How can I filter the data in the Transaction Tab to see only the items that interest me?

You’ll note that Store, Term and Employee data is presented in blue text (as is data in the ID column, but that column has unique characteristics which are discussed in the next sub-section).  By clicking on the data on any one of these columns, you can narrow the data set presented to include only those items that match the selected data value.  For example, if you click on an individual cell the Store column that contains the value 3247, then the data set will be narrowed to present only those transactions in the transaction log from Store 3247.  The data set can be narrowed in this fashion independent of the sort order.  You can continue to tweak the sort order of this smaller data set in the manner discussed in the “Sortability” sub-section.

Implementation Notes:

    • By clicking on a value in the ‘Term’ column, your resulting data set will be all those items in the transaction log from Store + Terminal.  For example, if you click on the Term value 0002 from a specific row, and the Store value in that row is 3247, the resulting data set will be all transactions in the database from Store + Term equal to 32470002.]

When you narrow the data set in this fashion, you are applying a filter to the data.  This filter is expressed in the upper right-hand corner of the Transaction tab screen in the “Search by” option.  Using the previous example, after you select 0002 form the Term column, you’ll notice that the “Search by” box is filled with the value 32470002.  To remove the filter, erase the value in the box – the entire data set will be re-displayed.

Other filters that can be applied are by card number (‘RAR’), Social Security Number (‘SSN’) and Store (which results in a data set that will consist of all transactions from all terminals at the selected store).

Alternatively, you can key-enter values directly into the “Search by” box.  This approach is the more logical one in situations where you’re looking to locate and isolate transactions that do not appear on current screen. 

Implementation Notes:

  • When key-entering data for Term values, you can use the “Search by” feature in two distinct ways:  entering a four-digit value (e.g., 0002) will return all transactions with that terminal ID across all stores; entering a eight-digit Store + Terminal value (e.g., 32470002) will return transactions for the terminal at that specific store.
  • When searching by Store values, providing a data value smaller than four digits will result in returning transactions from all stores that contain that initial string, e.g., entering 324 submits a query for 324*, meaning that the resulting data set would include all transactions from all terminals for stores 3240 through 3249.

1.1.5          How can I find out more about an individual transaction?

The blue text of the ID column is actually a hyperlink – by clicking on a specific ID, you can see all known information about the chosen transaction, arranged into four main data sections – Request Data, Response Data, Cardholder Data and Internal Data. 

If the configuration information for the terminal where the transaction originated from has a current profile where the audit capabilities are turned on, a fifth section called ‘Audit Data’ will be presented here.  This section contains a recording of the exact dialogue between OLS.Switch and the origination point.  The full request and reply are presented in Hex and ASCII formats.

Implementation Notes:

  • There is overhead involved with turning on Audit capabilities.  On busy systems, it should not be turned on en masse.  This problem-solving feature should be used in a targeted fashion to debug and troubleshoot specific issues.

Four fields – Store, Terminal, Name, SSN – provide links back to the main Transaction Tab display with data filtered by that value.  For example, if the value of Store in the individual transaction screen is equal to 3997, clicking on that value will return you to the list of transactions with a filter that shows only those transactions from that specific store.

1.2        Stores Tab

1.2.1          How is the stores data sorted?  Can I choose a different sort order?

By default, the information in the Stores tab is presented in order of ascending ID (the store number).  The current order of the data set is signified by a white arrow (ascending ▲; descending ▼) in the currently defined sort column.  By clicking on the appropriate column header, you can re-order the presentation data to be sorted by that column.  Currently, you can elect to sort the data by ID, Name, City and State values.  Clicking your chosen sort column a second time will change the direction of the sort from ascending to descending or vice versa.

1.2.2          How can I edit information about a store?

By clicking on any of the data fields associated with a specific store, you are presented with complete store information – all the fields associated with the store, as well as each of terminals set up as belonging to that specific store.  Click on the appropriate instance of “[Edit]” to update.

Implementation Notes: 

    • In implementations where OLS.Switch is not used to manage the ‘master’ list of stores and terminals, any change you make may be subsequently overlaid by a regularly scheduled refresh of the Store and Terminal data files.  In these situations, you should take the time to understand the flow and timing of inter-system processes to ensure that your changes stick.

1.2.3          How can I tell who made a change to a piece of store information?

All changes to store and terminal information are recorded in a revision history, which is presented beneath the terminal list for a store.

1.2.4          How do I turn on the audit capabilities for a specific terminal?

When the ‘audit’ feature is turned on for a terminal, OLS.Switch will record complete request and response images in the transaction log for transactions originating from that location.  This feature is helpful for debugging, testing and troubleshooting.  To turn auditing on, the ‘Profile’ field in the terminal definition must be set to a defined Terminal Profile within OLS.Switch. 

For example, imagine there are three values defined in the Terminal Profile table:

  • ACTIVE_NO_TRACE is a profile indicates a terminal is active and the audit is turned off
  • ACTIVE_TRACE is a profile indicates a terminal is active and the audit is turned on
  •  INACTIVE_NO_TRACE is a profile indicates a terminal is inactive and the audit is turned off

In this example, the audit facility can be turned on by setting the ‘Profile’ field to ACTIVE_TRACE. [Note that the names of the profiles used here are examples only and are left to the implementer of OLS.Switch to decide.]

1.2.5          What restrictions will I encounter in attempting to delete a store or terminal from OLS.Switch?

A store or terminal needs to be in an INACTIVE status (specifically, in the case of a terminal, needs to be associated with a terminal profile where the Active option is turned on) for a delete request to be accepted and processed by OLS.Switch.  Else, you’ll get a warning that you cannot delete an active store or terminal.

Assuming the store or terminal is in an inactive status, when you try to delete, then you should get a warning that advises you that when you delete this entry, all revision history will be lost and any existing transactions on the transaction log will be orphaned.  You’ll then be asked if you want to continue.

Implementation Notes:

    • In implementations where OLS.Switch is not used to manage the ‘master’ list of stores and terminals, any change you make (e.g., editing an address, deleting a store entry) may be subsequently overlaid by a regularly scheduled refresh of the Store and Terminal data files.  In these situations, you should take the time to understand the flow and timing of inter-system processes to ensure that your changes stick.
